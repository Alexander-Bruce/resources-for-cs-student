  ```
  declare @id int
  declare @name varchar(255)
  declare @msg varchar(255)
  
  set @id = 1
  set @name = 'Apple iPhone 15'
  
  if not exists (select * from product where id = @id)
      begin
          set @msg = N'没有该商品'
          print @msg
      end
  else
      begin
          update product set product_name = @name where id = @id
          select * from product where id = @id
      end
  ```


* 查询商品购买信息，将商品的购买数量都加1（提示：使用流程控制语句while）。

  ```
  declare @id int;
  
  declare product_order_info_cursor cursor for
  select id from order_info;
  open product_order_info_cursor;
  fetch next from product_order_info_cursor into @id;
  
  while @@FETCH_STATUS = 0
      begin
          update order_info
          set count = count + 1
          where id = @id;
          fetch next from product_order_info_cursor into @id;
      end
  close product_order_info_cursor;
  deallocate product_order_info_cursor;
  ```

* 定义一个用户自定义的函数，能够根据订单号，查询商品的购买数量，如果购买数量>2，输出订单号、商品名称和购买数量。

  ```
  create function GetProductDetails(@id varchar(255))
  returns table
  as
  return
  (
      select
          product_order.id,
          product.product_name,
          order_info.count
      from
          order_info
          join product_order on order_info.order_id = product_order.id
          join product on order_info.pid = product.id
      where
          order_info.order_id = @id
          and order_info.count > 2
  );
  ```

## 【任务总结】

1. 存储过程和函数的区别。

   #### 1. 定义和目的
   - **存储过程**：是一组预编译的 SQL 语句，可以接受参数并执行特定的任务。存储过程通常用于执行复杂的操作，如插入、更新、删除数据，或进行复杂的业务逻辑处理。
     
   - **函数**：通常用于计算并返回一个值，可以接受参数。函数通常用于数据处理、计算或转换，如在查询中进行计算或格式化。

   #### 2. 返回值
   - **存储过程**：可以返回多个值或结果集，但不一定需要返回值。通过输出参数或结果集返回信息。
     
   - **函数**：必须返回一个单一的值，可以是标量值、表值或其他数据类型，通常使用 `RETURN` 语句。

   #### 3. 调用方式
   - **存储过程**：通过 `EXECUTE` 或 `CALL` 语句来调用，不能在 SQL 语句中直接使用。
     
   - **函数**：可以在 SQL 语句中直接调用，就像内置函数一样。

   #### 4. 事务控制
   - **存储过程**：可以包含事务控制语句，用于处理复杂的事务逻辑。
     
   - **函数**：通常不允许使用事务控制语句，事务控制通常是在调用函数的上下文中进行的。

   #### 5. 使用场景
   - **存储过程**：适用于需要执行一系列操作、复杂业务逻辑或批处理任务的场景。
     
   - **函数**：适用于简单计算、转换或在查询中使用的场景。

   #### 总结
   ​	存储过程和函数各有用途，选择使用哪一种取决于具体的需求和上下文。在需要执行复杂操作时，可以选择存储过程，而在需要计算或转换值时，则可以使用函数。

2. DML触发器的工作原理。

​	DML（数据操纵语言）触发器是在数据库中定义的特殊类型的存储过程，用于自动响应对表的 DML 操作（如 `INSERT`、`UPDATE` 或 `DELETE`）。它们在特定事件发生时自动执行，以下是 DML 触发器的工作原理：

#### 1. 定义触发器
DML 触发器是在表上定义的，可以针对以下操作创建：
- **AFTER 触发器**：在 DML 操作完成后执行。
- **BEFORE 触发器**：在 DML 操作执行之前执行。

#### 2. 触发器的激活
​	当对特定表进行 DML 操作时，数据库管理系统会检查该表上是否有定义的触发器。如果有，触发器将自动被激活。具体过程如下：

- **INSERT 操作**：
  - 插入新记录后，如果表上有 `AFTER INSERT` 触发器，该触发器将执行。
  
- **UPDATE 操作**：
  - 更新记录后，如果表上有 `AFTER UPDATE` 触发器，该触发器将执行。
  
- **DELETE 操作**：
  - 删除记录后，如果表上有 `AFTER DELETE` 触发器，该触发器将执行。

#### 3. 使用虚拟表
在触发器内部，通常会使用两个虚拟表：
- **INSERTED**：用于存储新插入或更新后的记录。
- **DELETED**：用于存储被删除或更新前的记录。

这些虚拟表使得触发器可以访问与操作相关的旧数据和新数据。例如：
- 在 `UPDATE` 触发器中，`INSERTED` 表包含更新后的值，`DELETED` 表包含更新前的值。

#### 4. 执行触发器逻辑
​	当触发器被激活后，数据库执行触发器中的逻辑。触发器可以包括条件判断、数据验证、日志记录、触发其他操作等功能。

#### 5. 事务处理
​	触发器的执行是在 DML 操作的上下文中进行的，如果触发器中的逻辑导致错误，可能会导致整个 DML 操作回滚，从而保持数据的一致性。

### 6. 注意事项
- 触发器可能导致性能下降，特别是在高并发情况下。
- 触发器执行顺序可能会影响结果，特别是在存在多个触发器时。
- 触发器的使用需要谨慎，以避免复杂的依赖关系和难以调试的问题。

#### 总结
​	DML 触发器通过自动响应数据库操作，为数据完整性、审计和业务逻辑提供了一种强大机制。它们的使用可以增强应用程序的功能性和灵活性。
